# Feature Specification: تغليف البرنامج كملف تنفيذي مستقل

**Feature Branch**: `005-pkg-packaging`

**Created**: 2026-06-12

**Status**: Draft

**Input**: User description: "ابدأ التنفيذ pkg — تشغيل البرنامج عند العملاء بدون تثبيت Node.js، MySQL مثبت يدوياً"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - تشغيل البرنامج بدون تثبيت Node.js (Priority: P1)

صاحب المغسلة يحصل على ملف `laundry.exe` واحد، يضغط عليه مزدوجاً فيبدأ السيرفر ويفتح المتصفح على `localhost` — بدون أي خطوة تقنية إضافية.

**Why this priority**: هذا هو الهدف الجوهري للميزة: تصفير متطلبات التثبيت على جهاز العميل.

**Independent Test**: نقل `laundry.exe` لجهاز Windows نظيف (بدون Node.js) مع MySQL مثبت، تشغيله، والتحقق أن الواجهة تفتح في المتصفح بشكل طبيعي.

**Acceptance Scenarios**:

1. **Given** جهاز Windows 10+ بدون Node.js مع MySQL مثبت ومتاح، **When** المستخدم يشغّل `laundry.exe`، **Then** يبدأ السيرفر ويظهر المتصفح على شاشة تسجيل الدخول خلال 10 ثوانٍ.
2. **Given** `laundry.exe` يعمل بالفعل، **When** يحاول المستخدم تشغيله مرة ثانية، **Then** يظهر إشعار أن البرنامج يعمل بالفعل أو يُعاد توجيه المتصفح للصفحة المفتوحة.
3. **Given** MySQL متوقف أو إعدادات الاتصال خاطئة، **When** يُشغَّل `laundry.exe`، **Then** تظهر رسالة خطأ واضحة بالعربية تطلب مراجعة إعدادات قاعدة البيانات.

---

### User Story 2 - إعدادات الاتصال قابلة للتعديل (Priority: P2)

تقني التركيب يستطيع تعديل إعدادات اتصال MySQL (مضيف، مستخدم، كلمة مرور، اسم قاعدة البيانات) بدون إعادة بناء الملف التنفيذي.

**Why this priority**: كل عميل لديه إعدادات MySQL مختلفة؛ الـ `.exe` يجب أن يكون مرناً.

**Independent Test**: تعديل ملف `.env` بجانب `laundry.exe` وإعادة تشغيله للتحقق أنه يقرأ الإعدادات الجديدة.

**Acceptance Scenarios**:

1. **Given** ملف `.env` بجانب `laundry.exe` يحتوي بيانات MySQL صحيحة، **When** يُشغَّل `laundry.exe`، **Then** يتصل بقاعدة البيانات المحددة بنجاح.
2. **Given** لا يوجد ملف `.env`، **When** يُشغَّل `laundry.exe`، **Then** يستخدم القيم الافتراضية أو يظهر إرشاد لإنشاء الملف.

---

### User Story 3 - الملفات الحساسة والبيانات محفوظة خارج الـ exe (Priority: P2)

مجلدات `data/` و `ssl/` و ملف `.env` تبقى على القرص بجانب `laundry.exe` وليس مدمجة بداخله، بحيث يمكن تعديلها أو نسخها احتياطياً.

**Why this priority**: البيانات والجلسات والشهادات يجب أن تستمر بين إصدارات البرنامج ولا تُفقد عند تحديثه.

**Independent Test**: فحص أن `data/update-status.json` و `.env` يُكتبان ويُقرأان من مسار الـ exe وليس من داخله.

**Acceptance Scenarios**:

1. **Given** `laundry.exe` في مجلد `C:\Laundry\`، **When** يعمل البرنامج، **Then** ينشئ `C:\Laundry\data\` ويكتب إليه بشكل طبيعي.
2. **Given** تحديث `laundry.exe` بنسخة جديدة في نفس المجلد، **When** يعمل البرنامج، **Then** تبقى بيانات `data/` سليمة دون حذف.

---

### User Story 4 - تشغيل تلقائي مع Windows (Priority: P3)

البرنامج يبدأ تلقائياً عند تشغيل الجهاز دون تدخل المستخدم، مثل أي خدمة Windows.

**Why this priority**: المغسلة تحتاج البرنامج جاهزاً عند فتح الجهاز صباحاً بدون الضغط على أي شيء.

**Independent Test**: إعادة تشغيل الجهاز والتحقق أن البرنامج يبدأ تلقائياً خلال 60 ثانية من بدء Windows.

**Acceptance Scenarios**:

1. **Given** البرنامج مثبت كـ Windows Service، **When** يُعاد تشغيل الجهاز، **Then** يبدأ السيرفر تلقائياً ويكون جاهزاً خلال 60 ثانية.
2. **Given** السيرفر توقف بسبب خطأ، **When** تمر 5 ثوانٍ، **Then** يعيد Windows Service تشغيله تلقائياً.

---

### Edge Cases

- ماذا يحدث إذا كان البورت 3000 مشغولاً بعملية أخرى؟
- ماذا يحدث إذا كان مجلد `data/` للقراءة فقط (صلاحيات Windows)؟
- كيف يتعامل البرنامج مع مسارات تحتوي على أحرف عربية أو مسافات؟
- ماذا لو لم يكن Chrome/Chromium مثبتاً (مطلوب لطباعة الفواتير PDF)؟

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: يجب بناء الملف التنفيذي `laundry.exe` باستخدام `pkg` مع تضمين Node.js 18 x64.
- **FR-002**: يجب أن يقرأ البرنامج ملف `.env` من نفس مجلد `laundry.exe` (مسار مطلق عبر `process.execPath`).
- **FR-003**: يجب أن تُكتب وتُقرأ جميع ملفات `data/` من مسار `path.dirname(process.execPath)/data/` وليس `__dirname`.
- **FR-004**: يجب تضمين مجلدات `screens/`، `assets/` كـ assets ثابتة داخل الـ exe مع إتاحة `data/`، `.env`، `ssl/` خارجه.
- **FR-005**: يجب إنشاء سكريبت `scripts/install-service.ps1` لتسجيل البرنامج كـ Windows Service باستخدام `nssm`.
- **FR-006**: يجب أن تظهر رسالة خطأ عربية واضحة إذا فشل الاتصال بـ MySQL عند البدء.
- **FR-007**: يجب أن يُضاف أمر `npm run build` في `package.json` لبناء الـ exe.

### Key Entities

- **`laundry.exe`**: الملف التنفيذي المستقل الذي يحتوي Node.js runtime + كود السيرفر + الملفات الثابتة.
- **`process.execPath`**: مسار الـ exe الحالي — المرجع الأساسي لتحديد مسارات الملفات الخارجية.
- **`.env`** (خارجي): إعدادات قاعدة البيانات والبورت — يُعدَّل يدوياً لكل عميل.
- **`data/`** (خارجي): بيانات الجلسة، حالة التحديث، سجلات الأحداث.
- **Windows Service**: خدمة النظام التي تشغّل `laundry.exe` تلقائياً عند بدء Windows.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: يعمل البرنامج بشكل كامل على جهاز Windows 10+ بدون تثبيت Node.js مسبقاً.
- **SC-002**: يبدأ السيرفر ويكون جاهزاً لتلقي الطلبات خلال أقل من 15 ثانية من تشغيل `laundry.exe`.
- **SC-003**: جميع الوظائف الحالية (POS، فواتير، تقارير، WhatsApp) تعمل بنفس الأداء بعد التغليف.
- **SC-004**: حجم `laundry.exe` لا يتجاوز 100 MB.
- **SC-005**: تثبيت البرنامج كـ Windows Service لا يستغرق أكثر من 5 دقائق بمساعدة تقني.

## Assumptions

- MySQL مثبت يدوياً على جهاز العميل قبل تشغيل البرنامج — ليس مسؤولية هذه الميزة.
- الجهاز يعمل بـ Windows 10 أو أحدث (64-bit).
- Chrome/Chromium مثبت على جهاز العميل لأغراض طباعة PDF — خارج نطاق هذه الميزة.
- `nssm` (Non-Sucking Service Manager) يُوزَّع مع حزمة التثبيت لتسجيل الـ Windows Service.
- لا يوجد نظام installer (.exe setup) في هذه الميزة — التوزيع يكون بمجلد مضغوط (.zip) يحتوي `laundry.exe` + `.env` template + سكريبت التثبيت.
- نظام التحديث التلقائي (spec 004) يعمل بشكل مستقل ولا تتعارض معه مسارات الـ pkg.
