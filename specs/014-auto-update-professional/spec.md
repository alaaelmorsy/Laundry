# Feature Specification: نظام التحديث التلقائي الاحترافي

**Feature Branch**: `014-auto-update-professional`

**Created**: 2026-06-15

**Status**: Draft

**Input**: User description: "تنفيذ نظام تحديثات تلقائية مشابه لنظام cashier المُوثَّق في UPDATE_SYSTEM.md ولكن مُكيَّف مع برنامج الغسيل (pkg exe + Windows Service + invoke API)"

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 - فحص التحديثات يدوياً من الإعدادات (Priority: P1)

صاحب المغسلة يفتح شاشة الإعدادات ويضغط "فحص التحديثات"، تظهر نافذة (modal) تُخبره فوراً إذا كان هناك إصدار جديد مع رقم الإصدار، أو تُؤكد أنه يستخدم أحدث إصدار.

**Why this priority**: هذا المسار الأساسي الذي يبدأ منه كل تحديث. كل سيناريوهات التحديث تمر به أولاً.

**Independent Test**: فتح الإعدادات → ضغط "فحص التحديثات" → التحقق من ظهور النافذة ورقم الإصدار الجديد.

**Acceptance Scenarios**:

1. **Given** البرنامج يعمل وتوجد شبكة إنترنت، **When** المستخدم يضغط "فحص التحديثات" في الإعدادات، **Then** تفتح نافذة تعرض "جاري البحث عن التحديثات..." ثم تُظهر رقم الإصدار الجديد وزر "تحميل التحديث" خلال 15 ثانية.
2. **Given** البرنامج محدث على أحدث إصدار، **When** المستخدم يضغط "فحص التحديثات"، **Then** تظهر النافذة برسالة "✅ أنت تستخدم أحدث إصدار" بدون أزرار التحديث.
3. **Given** لا توجد شبكة إنترنت، **When** المستخدم يضغط "فحص التحديثات"، **Then** تظهر رسالة "❌ تعذّر الاتصال بالإنترنت — تحقق من الاتصال وأعد المحاولة".

---

### User Story 2 - تحميل التحديث مع شريط تقدم حقيقي (Priority: P1)

بعد أن يرى المستخدم أن هناك تحديثاً، يضغط "تحميل التحديث" ويرى شريط تقدم يتحرك فعلياً مع نسبة التحميل المئوية وسرعة التنزيل.

**Why this priority**: التحميل هو الخطوة الأكثر استهلاكاً للوقت، ويجب أن يرى المستخدم أن العملية تسير.

**Independent Test**: ضغط "تحميل التحديث" → مشاهدة شريط التقدم يتحرك → اكتمال التحميل وظهور زر "تثبيت الآن".

**Acceptance Scenarios**:

1. **Given** المستخدم ضغط "تحميل التحديث"، **When** يجري التنزيل، **Then** يُعرض شريط تقدم يتحرك من 0% إلى 100% مع الحجم المُنزَّل من الإجمالي بالـ MB.
2. **Given** انتهاء الدعم الفني (support_end_date مضى)، **When** يضغط "تحميل التحديث"، **Then** يظهر "⚠️ انتهت فترة الدعم الفني — يرجى تجديد الدعم للحصول على التحديثات" بدون تحميل.
3. **Given** اكتمل التحميل، **When** يرى المستخدم "✅ اكتمل التحميل!"، **Then** يظهر زر "تثبيت الآن" ويختفي زر "تحميل التحديث".

---

### User Story 3 - تثبيت التحديث وإعادة التشغيل (Priority: P1)

يضغط المستخدم "تثبيت الآن" فيُغلق البرنامج تلقائياً ويُستبدل الـ exe الجديد عبر `updater.ps1` ويعود البرنامج للعمل بالإصدار الجديد.

**Why this priority**: التثبيت هو الهدف النهائي — بدونه لا قيمة للتحميل.

**Independent Test**: ضغط "تثبيت الآن" → انتظار → فتح البرنامج → التحقق من رقم الإصدار الجديد في الإعدادات.

**Acceptance Scenarios**:

1. **Given** اكتمل تحميل التحديث، **When** يضغط المستخدم "تثبيت الآن"، **Then** يُغلق البرنامج فوراً ويبدأ `updater.ps1` بنسخ exe احتياطي ثم استبداله وإعادة تشغيل الـ service.
2. **Given** انتهاء الدعم الفني، **When** يضغط "تثبيت الآن"، **Then** يُرفض التثبيت مع رسالة "⚠️ انتهت فترة الدعم الفني".
3. **Given** فشل استبدال الـ exe، **When** يحدث الخطأ في `updater.ps1`، **Then** يُعاد تشغيل الإصدار القديم تلقائياً من النسخة الاحتياطية.

---

### User Story 4 - إشعار تلقائي عند فتح شاشة تسجيل الدخول (Priority: P2)

عند فتح البرنامج، تفحص شاشة تسجيل الدخول إن كان هناك تحديث متاح وتُظهر إشعاراً صغيراً يقترح الذهاب للإعدادات.

**Why this priority**: يُقلل احتمال بقاء العميل على إصدار قديم دون علمه.

**Independent Test**: نشر إصدار جديد على GitHub → إغلاق البرنامج وإعادة فتحه → التحقق من ظهور شريط الإشعار.

**Acceptance Scenarios**:

1. **Given** يوجد إصدار جديد متاح، **When** تُفتح شاشة تسجيل الدخول، **Then** يظهر شريط إشعار "🎉 يتوفر تحديث جديد! الإصدار X.X.XX — افتح الإعدادات للتحديث" يختفي بعد 6 ثوانٍ.
2. **Given** البرنامج محدث، **When** تُفتح شاشة تسجيل الدخول، **Then** لا يظهر أي إشعار.

---

### Edge Cases

- ماذا يحدث إذا انقطع الإنترنت أثناء تحميل الـ exe؟ → رسالة خطأ واضحة بالعربية، الملف المؤقت يُحذف، لا ضرر للبرنامج.
- ماذا يحدث إذا كان الـ exe المحمَّل تالفاً (checksum لا يطابق)؟ → يُرفض التثبيت مع رسالة "الملف تالف — أعد التحميل".
- ماذا يحدث إذا ضغط المستخدم "تحميل التحديث" مرتين؟ → يُتجاهل الطلب الثاني، التحميل يسير مرة واحدة.
- ماذا يحدث إذا كانت مساحة القرص غير كافية؟ → يُوقف التحميل مع رسالة "المساحة غير كافية".
- ماذا يحدث إذا كان `support_end_date` غير موجود في قاعدة البيانات؟ → يُعامَل كأن الدعم سارٍ (valid = true).
- ماذا يحدث إذا لم يُنشر الـ exe على GitHub Releases بعد؟ → رسالة "التحديث غير متاح للتنزيل حالياً" بدل خطأ تقني.

## Requirements *(mandatory)*

### Functional Requirements

**نظام الكشف عن التحديثات (updateService.js):**

- **FR-001**: يجب أن يتحقق البرنامج من GitHub Releases API (`/repos/{owner}/{repo}/releases/latest`) ويقارن رقم الإصدار الحالي بأحدث إصدار خلال 15 ثانية.
- **FR-002**: يجب تخزين نتيجة الفحص محلياً (cache لمدة ساعة) في `update-status.json` داخل مجلد بيانات البرنامج، لتجنب استهلاك GitHub API.
- **FR-003**: عند فشل الاتصال بـ GitHub، يُستخدم الـ cache المحلي إن وُجد ولم تنقضِ صلاحيته.
- **FR-004**: يجب أن تستخدم عملية التحميل تتبع التقدم الحقيقي (bytes received / total bytes) وترسلها للـ frontend عبر polling.

**التحقق من صلاحية الدعم الفني:**

- **FR-005**: قبل تحميل أو تثبيت التحديث، يجب التحقق من عمود `support_end_date` في جدول `app_settings` (صف id=1).
- **FR-006**: إذا كان `support_end_date` NULL أو غير موجود، يُعامَل كأن الدعم ساري (لا يُمنع التحديث).
- **FR-007**: إذا كان `support_end_date` في الماضي، يُمنع التحميل والتثبيت مع رسالة "انتهت فترة الدعم الفني".
- **FR-008**: فحص التحديثات (check only) مسموح دائماً بغض النظر عن حالة الدعم.

**API Endpoints (4-Step Checklist):**

- **FR-009**: إضافة دوال `checkForUpdates()`, `getSupportStatus()`, `startUpdateDownload()` في `database/db.js` أو `server/services/updateService.js`.
- **FR-010**: إضافة حالات `checkForUpdates`, `downloadUpdate`, `installUpdate`, `getUpdateProgress`, `getSupportStatus` في `server/invokeHandlers.js`.
- **FR-011**: تسجيل الـ methods في `assets/web-api.js` تحت `window.api`.
- **FR-012**: استدعاء الـ methods من `screens/settings/settings.js` و `screens/login/login.js`.

**واجهة نافذة التحديث في الإعدادات:**

- **FR-013**: يجب إضافة زر "فحص التحديثات" في شاشة الإعدادات يفتح نافذة modal.
- **FR-014**: تمر النافذة بحالات واضحة: checking → available/not-available → downloading (شريط تقدم) → downloaded → installing.
- **FR-015**: يجب أن تعرض النافذة رقم الإصدار الحالي ورقم الإصدار الجديد جنباً إلى جنب عند وجود تحديث.
- **FR-016**: خلال التحميل يجب عرض: النسبة المئوية، الحجم المُنزَّل / الإجمالي بالـ MB، وتقدير الوقت المتبقي.

**تنفيذ التثبيت (updater.ps1):**

- **FR-017**: يجب أن يستقبل `updater.ps1` المعاملات: `-ServiceName`, `-NewExePath`, `-CurrentExePath`.
- **FR-018**: يجب أن يُنشئ نسخة احتياطية من الـ exe الحالي قبل الاستبدال.
- **FR-019**: عند فشل الاستبدال يُعاد الـ exe القديم من النسخة الاحتياطية ويُعاد تشغيل الـ service.
- **FR-020**: يجب كتابة نتيجة التثبيت (نجاح/فشل) في `update-log.txt` قبل إعادة تشغيل الـ service.
- **FR-021**: يجب أن ينتظر الـ script على توقف الـ service حتى 30 ثانية ويُجبره على الإيقاف إن لم يتوقف.

**إشعار شاشة تسجيل الدخول:**

- **FR-022**: تفحص شاشة تسجيل الدخول الـ cache المحلي (`update-status.json`) بعد 2 ثانية من تحميلها.
- **FR-023**: إذا كان الـ cache يُشير لوجود تحديث، يظهر شريط إشعار يختفي تلقائياً بعد 6 ثوانٍ.

### Key Entities

- **update-status.json**: ملف cache محلي في مجلد البيانات يخزِّن `{ hasUpdate, latestVersion, downloadUrl, sha256Url, lastChecked, cachedUntil }`.
- **update-log.txt**: سجل تفصيلي لكل خطوات التحديث بطوابع زمنية في مجلد البيانات.
- **app_settings.support_end_date**: عمود DATE في جدول `app_settings` (صف id=1) يحدد تاريخ انتهاء الدعم الفني.
- **temp exe**: ملف الـ exe المؤقت الذي يُنزَّل `laundry-app-v{version}.exe` في مجلد التحميل المؤقت.
- **backup exe**: نسخة احتياطية من الـ exe الحالي يُنشئها `updater.ps1` قبل الاستبدال.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: عملية فحص التحديث تكتمل وتعرض النتيجة خلال 15 ثانية على شبكة عادية.
- **SC-002**: شريط التقدم أثناء التحميل يتحرك فعلياً (لا يظل ثابتاً) وتظهر النسبة المئوية الحقيقية.
- **SC-003**: 100% من حالات فشل استبدال الـ exe تنتهي بإعادة تشغيل الإصدار القديم بنجاح (لا يتعطل البرنامج).
- **SC-004**: لا تظهر أي رسالة تقنية بالإنجليزية للمستخدم — كل حالات الخطأ بالعربية ومفهومة.
- **SC-005**: بعد التحديث الناجح، يظهر رقم الإصدار الجديد في الإعدادات وجميع البيانات (فواتير، عملاء) سليمة.
- **SC-006**: منع التحميل/التثبيت يعمل صحيحاً عند انتهاء `support_end_date` ومسموح عند NULL أو تاريخ مستقبلي.

---

## Assumptions

- البرنامج يعمل كـ Windows Service عبر NSSM — اسم الـ service يُمرَّر كمعامل لـ `updater.ps1`.
- ملف الـ exe يُنشر على GitHub Releases باسم `laundry-app-v{version}.exe` مع ملف `sha256sums.txt`.
- `app_settings` يحتوي دائماً على صف واحد (id=1) — هذا مُثبَّت في architecture البرنامج.
- نظام التشغيل Windows 10/11 فقط.
- مجلد بيانات البرنامج (`data/`) قابل للكتابة من قِبَل الـ service.
- الـ polling للتقدم يحدث كل 500ms من الـ frontend.
- التحديث يدوي بالكامل — المستخدم يبدأ الفحص والتحميل والتثبيت يدوياً.
- فحص التحديث المؤجَّل (polling cache على login) لا يستدعي GitHub API مباشرةً — يقرأ الـ cache المحلي فقط.
