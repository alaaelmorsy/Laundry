# Feature Specification: Automatic Application Updates via GitHub Releases

**Feature Branch**: `004-auto-update-system`

**Created**: 2026-06-12

**Status**: Draft

**Input**: User description: "نظام تحديث تلقائي للبرنامج عبر GitHub Releases — زر فحص تحديثات في الإعدادات، إشعار بوجود تحديث في شاشة تسجيل الدخول، تحديث بضغطة واحدة مع نسخة احتياطية وتحديث قاعدة البيانات والتراجع التلقائي عند الفشل."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - تحديث البرنامج بضغطة واحدة من شاشة الإعدادات (Priority: P1)

صاحب المغسلة يفتح شاشة الإعدادات ويضغط زر "فحص التحديثات". إذا توفر إصدار جديد، يرى رقم الإصدار وملاحظاته، ثم يضغط "تحديث الآن" فيتم تنزيل التحديث والتحقق منه واستبدال الملفات وتحديث قاعدة البيانات وإعادة تشغيل التطبيق تلقائياً مع الحفاظ الكامل على بياناته وإعداداته.

**Why this priority**: هذا هو الهدف الأساسي للميزة — تمكين العميل من التحديث بدون أي تدخل تقني. بدونه لا قيمة للميزة.

**Independent Test**: نشر إصدار جديد على GitHub Releases ثم فتح الإعدادات والضغط على "فحص التحديثات" والتأكد من نجاح كامل دورة التحديث وعودة التطبيق للعمل بالإصدار الجديد مع بقاء جميع البيانات (الفواتير، العملاء، نقاط الولاء، الإعدادات) كما هي.

**Acceptance Scenarios**:

1. **Given** المستخدم في شاشة الإعدادات والإصدار الحالي هو الأحدث، **When** يضغط "فحص التحديثات"، **Then** تظهر رسالة "أنت تستخدم أحدث إصدار" مع عرض رقم الإصدار الحالي.
2. **Given** يوجد إصدار جديد منشور، **When** المستخدم يضغط "فحص التحديثات"، **Then** يعرض النظام رقم الإصدار الجديد وملاحظات الإصدار وزر "تحديث الآن".
3. **Given** المستخدم ضغط "تحديث الآن"، **When** يكتمل التحديث بنجاح، **Then** يتم: إنشاء نسخة احتياطية → تنزيل ملفات الإصدار → التحقق من سلامتها → إيقاف التطبيق → استبدال الملفات → تنفيذ ترقيات قاعدة البيانات → إعادة التشغيل → عرض رسالة نجاح مع رقم الإصدار الجديد.
4. **Given** اكتمل التحديث، **When** المستخدم يعيد فتح البرنامج، **Then** جميع بياناته السابقة (الفواتير، العملاء، الإعدادات، الجلسات) موجودة كما كانت.

---

### User Story 2 - إشعار بوجود تحديث في شاشة تسجيل الدخول (Priority: P2)

عند فتح البرنامج وظهور شاشة تسجيل الدخول، يفحص النظام في الخلفية وجود إصدار أحدث ويعرض إشعاراً واضحاً للمستخدم بوجود تحديث متاح، مع إمكانية الانتقال مباشرةً إلى شاشة التحديث.

**Why this priority**: يضمن أن العميل يعلم بالتحديثات دون الحاجة لتذكُّر الفحص اليدوي، لكنه يعتمد على وجود آلية التحديث الأساسية (P1).

**Independent Test**: نشر إصدار جديد ثم فتح البرنامج والوصول لشاشة تسجيل الدخول والتأكد من ظهور الإشعار خلال ثوانٍ، وأن الضغط عليه ينقل المستخدم إلى تدفق التحديث.

**Acceptance Scenarios**:

1. **Given** يوجد إصدار أحدث على GitHub، **When** تُعرض شاشة تسجيل الدخول، **Then** يظهر إشعار "تحديث جديد متاح — الإصدار X.Y.Z" بشكل غير مُعطِّل لتسجيل الدخول.
2. **Given** لا يوجد اتصال بالإنترنت أو فشل الفحص، **When** تُعرض شاشة تسجيل الدخول، **Then** لا يظهر أي إشعار ولا تتعطل عملية تسجيل الدخول.
3. **Given** المستخدم رأى الإشعار، **When** يضغط عليه، **Then** يُنقل إلى شاشة التحديث بعد تسجيل الدخول أو مباشرةً.

---

### User Story 3 - تراجع تلقائي عند فشل التحديث (Priority: P1)

إذا فشل أي جزء من عملية التحديث (تنزيل، تحقق سلامة، استبدال ملفات، ترقية قاعدة بيانات)، يقوم النظام تلقائياً باستعادة النسخة الاحتياطية ويعيد البرنامج إلى الحالة السابقة العاملة، مع رسالة واضحة للمستخدم تشرح ما حدث.

**Why this priority**: حماية بيانات العميل وضمان عدم تعطُّل عمله أهم من التحديث نفسه. أي خلل بدون تراجع قد يدمّر بيانات الإنتاج.

**Independent Test**: محاكاة فشل (مثلاً ملف تالف، انقطاع التنزيل، خطأ في migration) والتأكد من أن البرنامج يعود للإصدار السابق وقاعدة البيانات تعود لحالتها قبل التحديث وأن المستخدم يستطيع المتابعة فوراً.

**Acceptance Scenarios**:

1. **Given** فشل التحقق من سلامة الملفات المنزّلة، **When** يكتشف النظام ذلك، **Then** يُلغى التحديث قبل المساس بأي ملف ويعرض رسالة "فشل التحقق من سلامة التحديث".
2. **Given** فشل migration قاعدة البيانات في منتصف التنفيذ، **When** يلتقط النظام الخطأ، **Then** يستعيد ملفات الإصدار السابق ونسخة قاعدة البيانات الاحتياطية ويعيد التشغيل على الإصدار القديم.
3. **Given** نجح التراجع، **When** المستخدم يستخدم البرنامج، **Then** يعمل بنفس الإصدار السابق ولا يفقد أي بيانات أُنشئت قبل محاولة التحديث.

---

### Edge Cases

- **انقطاع الإنترنت أثناء التنزيل**: يلغى التحديث، لا تُمسّ ملفات الإنتاج، يُسمح بإعادة المحاولة.
- **مساحة قرص غير كافية للنسخة الاحتياطية أو التنزيل**: يُمنع بدء التحديث وتُعرض رسالة واضحة بالمساحة المطلوبة.
- **محاولة المستخدم إغلاق البرنامج أثناء التحديث**: يُحذَّر ويُمنع الإغلاق حتى انتهاء العملية أو التراجع.
- **تشغيل أكثر من نسخة من البرنامج**: يجب اكتشاف ذلك وإيقافها قبل بدء استبدال الملفات.
- **إصدار GitHub بدون ملاحظات أو بأصول ناقصة**: يُعتبر غير صالح ويُتجاهل مع تسجيل سبب التجاهل.
- **migration قاعدة بيانات سبق تنفيذه**: لا يُعاد تنفيذه (idempotency).
- **الضغط المتكرر على "فحص التحديثات"**: يُحدّ معدل الطلبات لتفادي تجاوز سقف GitHub API.
- **التحديث من إصدار قديم جداً يتطلب عدة migrations متتابعة**: يجب تنفيذها بالترتيب الصحيح أو رفض القفز ومطالبة المستخدم بالتحديث للإصدار الوسيط أولاً.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: يجب أن يوفر النظام زر "فحص التحديثات" في شاشة الإعدادات يقارن الإصدار المثبَّت بأحدث إصدار منشور على GitHub Releases.
- **FR-002**: عند عدم وجود تحديث، يجب أن يعرض النظام رسالة "أنت تستخدم أحدث إصدار" مع رقم الإصدار الحالي.
- **FR-003**: عند وجود تحديث، يجب أن يعرض النظام رقم الإصدار الجديد، ملاحظات الإصدار (release notes)، وزر "تحديث الآن".
- **FR-004**: يجب أن يفحص النظام تلقائياً وجود إصدار جديد عند عرض شاشة تسجيل الدخول ويعرض إشعاراً غير مُعطِّل عند توفره.
- **FR-005**: قبل أي تعديل، يجب أن ينشئ النظام نسخة احتياطية كاملة من قاعدة البيانات ومن ملفات الإعدادات وأي بيانات قابلة للتغير (جلسة WhatsApp، الإعدادات، الملفات المرفوعة).
- **FR-006**: يجب أن يتحقق النظام من سلامة ملفات التحديث المنزّلة (checksum/توقيع) قبل البدء في استبدال أي ملف.
- **FR-007**: يجب أن يوقف النظام التطبيق بأمان قبل استبدال الملفات وأن يضمن عدم وجود عمليات كتابة جارية على قاعدة البيانات.
- **FR-008**: يجب أن ينفّذ النظام ترقيات قاعدة البيانات (migrations) المطلوبة للإصدار الجديد بشكل آمن وقابل للتكرار (idempotent).
- **FR-009**: يجب أن يعيد النظام تشغيل التطبيق تلقائياً بعد نجاح التحديث ويعرض رسالة نجاح مع رقم الإصدار الجديد.
- **FR-010**: في حال فشل أي خطوة (تنزيل، تحقق، استبدال، migration)، يجب أن يستعيد النظام تلقائياً النسخة الاحتياطية ويعيد البرنامج للإصدار السابق.
- **FR-011**: يجب أن يحافظ التحديث على جميع بيانات المستخدم: قاعدة البيانات، الإعدادات، جلسة WhatsApp، الفواتير، العملاء، نقاط الولاء، التقارير، والملفات المخصصة.
- **FR-012**: يجب أن يعرض النظام تقدّم العملية للمستخدم (مرحلة حالية ونسبة إنجاز) خلال التنزيل والاستبدال.
- **FR-013**: يجب أن يمنع النظام إغلاق التطبيق أو بدء تحديث ثانٍ أثناء جريان تحديث.
- **FR-014**: يجب أن يتحقق النظام قبل البدء من توفر مساحة قرص كافية للنسخة الاحتياطية وللتحديث ويرفض البدء بدونها مع رسالة واضحة.
- **FR-015**: يجب أن يسجّل النظام أحداث التحديث (بدء، نجاح، فشل، تراجع) مع الطوابع الزمنية لتسهيل التشخيص لاحقاً.
- **FR-016**: يجب أن يتعامل النظام بهدوء مع انعدام الاتصال بالإنترنت أو فشل GitHub API دون التأثير على عمل البرنامج العادي.
- **FR-017**: يجب أن يحترم النظام حدود معدل GitHub API ولا يتسبب في حظر العميل.
- **FR-018**: يجب أن يعرض الإصدار الحالي للمستخدم بوضوح في شاشة الإعدادات حتى عند عدم وجود تحديث.

### Key Entities *(include if feature involves data)*

- **Release (إصدار)**: يمثّل إصداراً منشوراً على GitHub، يحوي رقم الإصدار، ملاحظات الإصدار، وأصول التنزيل (الملفات) ومجموعات التحقق.
- **Update Session (جلسة تحديث)**: عملية تحديث واحدة لها حالة (تنزيل، تحقق، استبدال، migration، نجاح، فشل، تراجع) ووقت بدء وانتهاء وسجل أحداث.
- **Backup (نسخة احتياطية)**: لقطة محفوظة لقاعدة البيانات والإعدادات والملفات الحساسة قبل التحديث، مع رقم الإصدار المرجعي وتاريخ الإنشاء.
- **Migration (ترقية قاعدة بيانات)**: تغيير على بنية أو بيانات قاعدة البيانات مرتبط بإصدار محدد، يجب تنفيذه مرة واحدة بأمان.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: يستطيع المستخدم إكمال عملية التحديث بأقل من 5 ضغطات (فحص → عرض → تحديث الآن → تأكيد → إعادة تشغيل).
- **SC-002**: 99% من محاولات التحديث الناجحة لا تتسبب في فقدان أي سجل من بيانات المستخدم.
- **SC-003**: في كل حالة فشل، يعود البرنامج للإصدار السابق العامل خلال أقل من دقيقتين دون تدخل المستخدم.
- **SC-004**: يظهر إشعار وجود تحديث في شاشة تسجيل الدخول خلال 5 ثوانٍ من فتح البرنامج عند توفر اتصال إنترنت.
- **SC-005**: 0% من محاولات التحديث تستبدل ملفات إنتاج بدون التحقق المسبق من سلامة الملفات المنزّلة.
- **SC-006**: العميل قادر على إجراء التحديث بدون أي تواصل مع الدعم الفني في 95% من الحالات.
- **SC-007**: لا يؤثر فحص التحديثات في الخلفية على زمن ظهور شاشة تسجيل الدخول بأكثر من 500 مللي ثانية.

## Assumptions

- يستخدم النظام GitHub Releases للمستودع الرسمي للبرنامج كمصدر وحيد للتحديثات الموثوقة.
- العميل يملك اتصال إنترنت لحظة التحديث (وليس بالضرورة بشكل دائم).
- البرنامج يعمل على جهاز Windows واحد لكل ترخيص؛ سيناريوهات الشبكة متعددة الأجهزة خارج نطاق هذه النسخة.
- النسخ الاحتياطية تُحفظ محلياً على جهاز العميل (تخزين سحابي خارج النطاق).
- ترقيات قاعدة البيانات تأتي ضمن أصول الإصدار وتُنفَّذ بنفس آلية الترقية المعتمدة في المشروع.
- لا يحتاج المستخدم لصلاحيات أعلى من تشغيل التطبيق العادي لإجراء التحديث (مكان التثبيت قابل للكتابة من قبل المستخدم).
- الحفاظ على ملفات `data/whatsapp_session/` و قاعدة البيانات والإعدادات المخصصة جزء أساسي من التحديث.
