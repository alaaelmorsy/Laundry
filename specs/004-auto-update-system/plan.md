# Implementation Plan: Automatic Application Updates via GitHub Releases

**Branch**: `004-auto-update-system` | **Date**: 2026-06-12 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/004-auto-update-system/spec.md`

## Summary

نظام تحديث تلقائي يتيح لصاحب المغسلة تحديث البرنامج بضغطة واحدة من شاشة الإعدادات، مع إشعار تلقائي في شاشة تسجيل الدخول عند توفر إصدار جديد. يعتمد على GitHub Releases كمصدر للتحديثات، ويستخدم سكريبت PowerShell منفصل لاستبدال الملفات بعد إغلاق السيرفر، مع نسخة احتياطية كاملة وتراجع تلقائي عند الفشل.

## Technical Context

**Language/Version**: Node.js 18+ (CommonJS), PowerShell 5.1 (Windows built-in)

**Primary Dependencies**: `https` (built-in), `crypto` (built-in), `fs` (built-in), `child_process` (built-in) — no new npm packages required. `node-fetch` NOT used; native `https.get` for GitHub API calls.

**Storage**: `data/update-status.json` (cached check result), `data/update-log.txt` (event log), `backup/pre-{version}/` (pre-update snapshot)

**Testing**: Manual validation per `quickstart.md` scenarios. No automated test framework in project.

**Target Platform**: Windows 10+ (local on-premise installation)

**Project Type**: Desktop web app (local Express server + browser frontend)

**Performance Goals**: Update check ≤ 2s on normal connection; login screen notification appears within 5s; download progress shown in real-time.

**Constraints**: No new npm dependencies; PowerShell 5.1 only (no PS7); must not overwrite `data/`, `.env`, `ssl/`; idempotent DB migrations only; backup before any file replacement.

**Scale/Scope**: Single installation per laundry. Update affects one server process. No concurrency concerns.

## Implementation Phases

### Phase A: Core Update Service (Backend)

**Files**: `server/services/updateService.js`, `migrate.js`, `scripts/updater.ps1`

Responsibilities:
1. `updateService.js`:
   - `checkForUpdate(force)` — hits GitHub API, writes `data/update-status.json`, returns comparison
   - `getUpdateStatus()` — reads `data/update-status.json` only
   - `getUpdateProgress()` — returns in-memory progress state
   - `performUpdate()` — pre-flight checks (disk space, no active session), then:
     a. Create `backup/pre-{version}/` with source snapshot + DB dump (`mysqldump`)
     b. Download ZIP to temp file with progress tracking
     c. Verify SHA256 checksum
     d. Write `data/update-status.json` with `status: "in_progress"`
     e. Spawn `scripts/updater.ps1` (detached) with: PID, version, zip path, backup path
     f. Schedule server graceful exit in 1 second

2. `migrate.js` — calls `db.initialize()` then exits. Used by `updater.ps1` after file replacement.

3. `scripts/updater.ps1`:
   - Waits for server PID to exit (poll every 500ms, timeout 30s)
   - Extracts ZIP to app root (skip `data/`, `.env`, `ssl/`)
   - Runs `node migrate.js`; on failure → restore from backup, run old `node migrate.js`
   - Writes result to `data/update-status.json`
   - Starts server: `Start-Process node -ArgumentList "server/index.js"`
   - Logs all steps to `data/update-log.txt`

### Phase B: API Registration (4-Step Checklist)

**Files**: `database/db.js`, `server/invokeHandlers.js`, `assets/web-api.js`

For each of the 4 methods (`checkForUpdate`, `getUpdateStatus`, `performUpdate`, `getUpdateProgress`):
1. `db.js` — no DB functions needed; skip step 1
2. `invokeHandlers.js` — add `case 'checkForUpdate':`, `case 'getUpdateStatus':`, etc., each wrapping `updateService` calls in `try/catch`
3. `web-api.js` — register 4 methods under `window.api`
4. Screen JS — called from Settings and Login screens (Phase C)

### Phase C: Settings Screen — Update Tab

**Files**: `screens/settings/settings.html`, `screens/settings/settings.js`

New tab "التحديثات" (tab ID `tabUpdate`, panel ID `panelUpdate`) added alongside existing tabs.

**Panel layout**:
- Current version display: `الإصدار الحالي: 1.0.12`
- "فحص التحديثات" button (`btnCheckUpdate`)
- Result area: either "أنت تستخدم أحدث إصدار" or update card with version, release notes (scrollable), and "تحديث الآن" button
- Progress panel (hidden until update starts): step list with icons + percentage bar
- On update start: disable "تحديث الآن", show progress panel, poll `getUpdateProgress` every 1s
- On server disconnect (fetch fails): show "جارٍ إعادة التشغيل... يُرجى الانتظار" and poll `getUpdateStatus` every 3s until reconnected

### Phase D: Login Screen — Update Notification

**Files**: `screens/login/login.html`, `screens/login/login.js`

On `DOMContentLoaded`, call `getUpdateStatus()` (reads cache, fast):
- If `hasUpdate: true`: show a non-blocking banner at the top of the login card
  - Text: `تحديث جديد متاح — الإصدار {latestVersion}`
  - Link/button: "انتقل إلى الإعدادات بعد تسجيل الدخول"
- If no update or error: show nothing (login form unaffected)

Also trigger a background `checkForUpdate({ force: false })` after 2s to refresh the cache if it is > 60 minutes old (fire-and-forget, errors silently ignored).

## Risk & Mitigations

| Risk | Mitigation |
|------|-----------|
| PowerShell script fails to start | Pre-check `$PSVersionTable` before spawning; fall back to `.bat` wrapper if needed |
| `mysqldump` not on PATH | Try `mysqldump`; if not found, use Node.js pool to export all tables as INSERT statements |
| GitHub API rate limit (60/hr anon) | Cache result 60 min; use `If-None-Match` with ETag from last response |
| ZIP extraction overwrites wrong files | Explicit allowlist of paths to extract; never extract `data/`, `.env`, `ssl/` |
| Server port still in use when updater tries to restart | PowerShell: `netstat` check + 5s wait before starting `node server/index.js` |
| Update fails after file replacement but before migration | Restore from source backup; re-run old `migrate.js` against restored DB backup |
