# Feature Specification: إصلاح شامل لنظام التحديث التلقائي

**Feature Branch**: `013-fix-update-system-professional`

**Created**: 2026-06-15

**Status**: Draft

**Input**: User description: "اريد حل جميع مشاكل التحديثات فى البرنامج راجع عليها كلها واجعلها احترافية وبدون اى مشاكل علما ان البرنامج هيكون ملف exe"

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 - تحديث ناجح من البداية للنهاية (Priority: P1)

صاحب المغسلة يفتح شاشة الإعدادات ويضغط "فحص التحديثات"، يرى الإصدار الجديد، يضغط "تحديث الآن"، ويرى شريط تقدم واضحاً لكل خطوة (نسخة احتياطية ← تحميل ← تحقق ← استبدال ← إعادة تشغيل). بعد انتهاء التحديث يعود البرنامج تلقائياً بإصدار جديد وبيانات سليمة 100%.

**Why this priority**: هذا هو الهدف الكامل. أي خطأ في هذا المسار يجعل العميل يحتاج دعماً تقنياً.

**Independent Test**: رفع إصدار جديد على GitHub Releases → فتح الإعدادات → فحص → تحديث → التحقق من رقم الإصدار الجديد والبيانات الموجودة.

**Acceptance Scenarios**:

1. **Given** البرنامج يعمل وتوجد شبكة إنترنت، **When** المستخدم يضغط "فحص التحديثات"، **Then** يظهر رقم الإصدار الجديد وملاحظاته وحجم الملف خلال 15 ثانية.
2. **Given** المستخدم ضغط "تحديث الآن"، **When** تبدأ العملية، **Then** يظهر شريط تقدم لكل خطوة منفصلة يمتلئ تدريجياً من 0% إلى 100%.
3. **Given** اكتملت عملية التحديث، **When** يعود البرنامج للعمل، **Then** رقم الإصدار في الإعدادات يطابق الإصدار الجديد وجميع البيانات (فواتير، عملاء، إعدادات) موجودة.
4. **Given** التحديث اكتمل، **When** المستخدم يضغط "فحص التحديثات" مرة ثانية، **Then** تظهر رسالة "أنت تستخدم أحدث إصدار".

---

### User Story 2 - معالجة الأخطاء مع رسائل واضحة (Priority: P2)

عند حدوث أي خطأ أثناء التحديث (انقطاع الإنترنت، timeout، فشل استبدال الملف) يرى المستخدم رسالة خطأ واضحة بالعربية تشرح المشكلة، ويستطيع إعادة المحاولة.

**Why this priority**: البرنامج يعمل على أجهزة العملاء — الشبكة غير مضمونة دائماً.

**Independent Test**: قطع الإنترنت أثناء التحميل والتحقق من ظهور رسالة عربية واضحة وإمكانية إعادة المحاولة.

**Acceptance Scenarios**:

1. **Given** فشل تحميل ملف التحديث بسبب timeout، **When** يحدث الخطأ، **Then** يظهر شريط بالأحمر ورسالة "فشل التنزيل — تحقق من الإنترنت وأعد المحاولة" وزر للمحاولة مجدداً.
2. **Given** تعذّر التحقق من الـ checksum بسبب مشكلة شبكة، **When** يحدث الخطأ، **Then** يكمل البرنامج التحديث مع تسجيل تحذير (التحقق اختياري عند فشل تحميل ملف الـ checksum).
3. **Given** فشل استبدال الـ exe بسبب قفل الملف، **When** يحدث الخطأ، **Then** يُعاد تشغيل الإصدار القديم تلقائياً ويرى المستخدم رسالة "فشل التحديث — تم الإبقاء على الإصدار القديم".

---

### User Story 3 - إشعار التحديث عند فتح البرنامج (Priority: P3)

عندما يكون هناك إصدار جديد متاح، يظهر إشعار صغير في شاشة تسجيل الدخول يخبر المستخدم بوجود تحديث.

**Why this priority**: يقلل احتمال بقاء العميل على إصدار قديم لفترة طويلة.

**Independent Test**: نشر إصدار جديد على GitHub → إغلاق البرنامج وإعادة فتحه → التحقق من ظهور إشعار التحديث.

**Acceptance Scenarios**:

1. **Given** يوجد إصدار جديد، **When** تُفتح شاشة تسجيل الدخول، **Then** يظهر شريط إشعار صغير "يوجد تحديث جديد — اذهب للإعدادات".
2. **Given** البرنامج محدث، **When** تُفتح شاشة تسجيل الدخول، **Then** لا يظهر أي إشعار.

---

### Edge Cases

- ماذا يحدث إذا انقطع الإنترنت أثناء تحميل ملف الـ exe؟ → رسالة خطأ واضحة، الملف المؤقت يُحذف، لا ضرر للبرنامج القائم.
- ماذا يحدث إذا كانت مساحة القرص غير كافية؟ → يُوقف التحديث قبل البدء برسالة "المساحة غير كافية".
- ماذا يحدث إذا ضغط المستخدم "تحديث الآن" مرتين؟ → يُتجاهل الطلب الثاني، التحديث يعمل مرة واحدة فقط.
- ماذا يحدث إذا فشل استبدال الـ exe لأن الملف مقفول؟ → يُعاد تشغيل الإصدار القديم والبيانات سليمة.
- ماذا يحدث إذا كانت ملفات GitHub Releases غير موجودة (لم يُرفع الـ exe)؟ → تظهر رسالة "التحديث غير متاح للتنزيل حالياً" بدل خطأ فني.
- ماذا يحدث إذا لم يكن اتصال بـ GitHub (جهاز offline)؟ → رسالة واضحة "تعذّر الاتصال بالإنترنت" بدل timeout فني.

## Requirements *(mandatory)*

### Functional Requirements

**نظام الكشف عن التحديثات:**
- **FR-001**: عند الضغط على "فحص التحديثات" يجب أن يتواصل البرنامج مع GitHub Releases API ويقارن رقم الإصدار الحالي بأحدث إصدار منشور خلال 15 ثانية كحد أقصى.
- **FR-002**: يجب أن تدعم عملية الفحص الـ redirects تلقائياً (GitHub يُعيد توجيه روابط التحميل).
- **FR-003**: يجب أن يُخزَّن نتيجة الفحص محلياً (cache ساعة واحدة) لتجنب استهلاك GitHub API rate limit.
- **FR-004**: عند فشل الاتصال بـ GitHub، يُعاد استخدام الـ cache المحلي إن وُجد، ويُعرض آخر تاريخ فحص.

**واجهة شريط التقدم:**
- **FR-005**: يجب أن يكون لكل خطوة من خطوات التحديث (نسخة احتياطية، تحميل، تحقق، استبدال، إعادة تشغيل) شريط تقدم منفصل يمتلئ من 0% إلى 100%.
- **FR-006**: خطوة التحميل يجب أن تعرض التقدم الحقيقي بالـ bytes المُنزَّلة من إجمالي حجم الملف.
- **FR-007**: الخطوات التي لا يمكن قياسها بدقة (نسخة احتياطية، تحقق، استبدال) تُعرض بـ animation تلقائية تُقدِّر الوقت المتوقع.
- **FR-008**: عند الخطأ يتحول شريط التقدم للأحمر مع رسالة عربية واضحة للمشكلة.

**تنفيذ التحديث (updater.ps1):**
- **FR-009**: يجب تمرير رقم الإصدار الحالي (fromVersion) كمعامل لـ updater.ps1 من داخل البرنامج (لا قراءة package.json من القرص — غير موجود في وضع exe).
- **FR-010**: يجب تمرير اسم الـ Windows Service كمعامل (لا hardcode) لدعم بيئات تثبيت مختلفة.
- **FR-011**: يجب أن ينتظر updater.ps1 على انتهاء عملية الـ server حتى 30 ثانية ويقتل العملية إن لم تخرج.
- **FR-012**: يجب إنشاء نسخة احتياطية من الـ exe القديم قبل استبداله.
- **FR-013**: عند فشل الاستبدال يُعاد الـ exe القديم من النسخة الاحتياطية تلقائياً.
- **FR-014**: بعد الاستبدال الناجح يُحذف ملف الـ exe المؤقت المحمَّل.
- **FR-015**: يُكتب نتيجة التحديث (نجاح أو rollback) في ملف الحالة قبل إعادة تشغيل الـ service.

**عملية البناء والنشر (GitHub Actions):**
- **FR-016**: يجب أن يُبنى الـ exe ويُرفع للـ Release باسم `laundry-app-v{version}.exe`.
- **FR-017**: يجب أن يُرفع `sha256sums.txt` يحتوي على hash الـ update exe والـ installer.
- **FR-018**: يجب أن يُنتَج الـ installer (Inno Setup exe) ويُرفع للـ Release.

**إشعار التحديث:**
- **FR-019**: يجب أن تفحص شاشة تسجيل الدخول عند الفتح إن كان هناك تحديث متاح (من الـ cache المحلي) وتعرض إشعاراً.

### Key Entities

- **update-status.json**: ملف محلي يخزِّن حالة الفحص (hasUpdate، latestVersion، downloadUrl، checksumUrl، lastChecked، lastUpdateResult)
- **update-log.txt**: سجل تفصيلي لكل خطوات التحديث بطوابع زمنية
- **temp exe**: الملف المؤقت الذي يُنزَّل في `{data}/laundry-app-v{version}.exe`
- **service name**: اسم الـ Windows Service الذي يُدير تشغيل البرنامج

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: عملية التحديث الكاملة من ضغط "تحديث الآن" حتى عودة البرنامج تستغرق أقل من 5 دقائق على إنترنت بسرعة 10 Mbps.
- **SC-002**: لا يُفقد أي سجل من قاعدة البيانات (فواتير، عملاء، إعدادات) بعد التحديث الناجح.
- **SC-003**: 100% من حالات فشل استبدال الـ exe تنتهي بإعادة تشغيل الإصدار القديم بنجاح.
- **SC-004**: جميع رسائل الخطأ التي تظهر للمستخدم تكون بالعربية وتشرح المشكلة بوضوح (لا رسائل تقنية بالإنجليزية).
- **SC-005**: شريط التقدم يعرض حركة حقيقية (لا يبقى ثابتاً) خلال كل خطوة من خطوات التحديث.
- **SC-006**: الكشف عن التحديثات يعمل على جهاز متصل بالإنترنت خلال 10 ثوانٍ.

## Assumptions

- البرنامج مُثبَّت كـ Windows Service باسم `LaundryPlusApp` عبر NSSM — هذا الاسم يجب أن يكون مُمرَّراً كمعامل.
- ملف `package.json` غير موجود على القرص في وضع exe (مدمج في الـ virtual FS) — كل المعلومات المطلوبة للـ updater تُمرَّر كمعاملات.
- الـ exe يكون دائماً في نفس مجلد بيانات البرنامج (data، ssl، .env).
- المستخدم لديه صلاحيات كافية لاستبدال ملف الـ exe (Windows Service تعمل كـ administrator).
- الشبكة قد تكون بطيئة أو غير مستقرة — timeout كافٍ (120 ثانية للتحميل، 15 ثانية للـ API).
- GitHub Releases API متاحة (قد تكون محظورة في بعض الشبكات — حالة طبيعية للفشل).
- اسم ملف الـ update exe دائماً `laundry-app-v{version}.exe`.
- نظام التشغيل Windows 10/11 فقط.
